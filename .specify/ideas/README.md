# 想法池 (Idea Backlog)

**用途**: 记录尚未成熟的想法，在创建正式规格前进行快速收集和整理。

---

## 概述

想法池是 spec-kit 工作流中的**第一阶段**，用于在创建完整规格之前快速记录和管理想法。

```
想法池 → 规格 → 计划 → 任务 → 实现
```

## 使用方法

### 添加新想法

```bash
# 简单添加 (默认重要度2)
/speckit.idea 添加一个用户认证功能

# 指定重要度
/speckit.idea 重要度3 实现实时通知功能
```

### 澄清想法 (Clarify)

澄清是将笼统想法细化为具体需求模块的关键步骤：

```bash
/speckit.idea clarify "想法标题"
```

**澄清流程**：
1. 依次询问 6 个关键问题（问题域、目标用户、核心价值、范围边界、依赖关系）
2. 分析回答，判断想法是否需要拆分
3. **如需拆分**：分析功能边界，按层次分组（数据层、采集层、分析层、模型层等），输出模块列表和依赖关系图
4. **无需拆分**：直接完善想法内容
5. 更新想法文件，归档澄清结果
6. 设置重要度：1、2、3，3最重要

**何时需要拆分**：
- 想法包含多个不同类型的功能
- 单次实现风险过高
- 需要分阶段交付
- 有多个团队可以并行开发

### 查看所有想法

```bash
/speckit.idea list
```

### 升级想法为规格 (ToSpec)

```bash
/speckit.idea tospec "想法标题"
```

当想法成熟后，将其升级为完整的规格文档：

**tospec 做什么**：
1. 读取想法文件的所有内容（描述、澄清答案、拆分模块等）
2. 调用 `/speckit.specify`
3. 创建 `specs/xxx-feature/` 目录
4. 生成 `spec.md`（包含需求、API设计、数据模型、验收标准等）
5. 想法状态改为 ✅已完成

**已拆分的想法**：
- 升级整个想法 → 生成包含所有模块的大规格
- 升级单个模块 → 为该模块创建独立规格

### 设置重要度

```bash
/speckit.idea set-priority "想法标题" 重要度3
```

## 文件结构

```
.specify/ideas/
├── index.md               # 想法索引导航
├── idea-template.md       # 想法模板
├── 想法名称.md            # 具体想法文件 (使用中文命名)
└── ...
```

## 重要度说明

| 重要度 | 含义 | 图标 |
|--------|------|------|
| 3 | 最重要 - 核心功能，必须做 | 🔥 |
| 2 | 中等重要 - 正常功能，按计划做 | 📋 |
| 1 | 不重要 - 有空做，暂不考虑 | 💤 |

## 状态说明

| 状态 | 含义 | 图标 |
|------|------|------|
| 待整理 | 新创建的想法，尚未审核 | 📝 |
| 已确认 | 已澄清，等待升级为规格 | 💡 |
| 已完成 | 已升级为规格或已实现 | ✅ |

## 与 spec-kit 工作流的集成

**整体流程**：
```
idea → spec → plan → tasks → implement
```

**idea 阶段内部流程**：
```
/speckit.idea add "描述"
    ↓ 创建 .specify/ideas/想法名称.md
/speckit.idea clarify "想法标题"
    ↓ 询问5个问题，分析并拆分模块（如需）
/speckit.idea tospec "想法标题" 或 "模块名称"
    ↓ 调用 /speckit.specify，进入 spec 阶段
```

## 最佳实践

1. **快速记录** - 不要追求完美，先记录下来
2. **澄清必做** - 使用 `clarify` 将笼统想法细化为具体需求
3. **按需拆分** - 复杂想法拆分后可逐个模块 tospec
4. **设置重要度** - 帮助确定工作顺序
5. **关联想法** - 在"相关链接"中引用相关的想法或规格
